# Dice Window: ТЗ, План и Модуль-Сервис

## 1) Цель
Добавить отдельное окно для бросков дайсов DnD, открываемое из лаунчера, с красивым визуальным интерфейсом, точной механикой бросков и изоляцией логики от остальных окон.

## 2) Финальное ТЗ

### 2.1 Точка входа
- В лаунчере есть кнопка `Дайсы`.
- По нажатию открывается отдельное окно дайсов.

### 2.2 Окно
- Начальный размер: `300x400`.
- Окно можно расширять.
- Always-on-top не требуется.
- При недостатке места используется вертикальная прокрутка.

### 2.3 Блоки UI
Сверху вниз:
1. Крупное поле результатов.
2. Блок `D20`.
3. Блок `D4, D6, D8, D10, D12`.
4. Блок `D100`.
5. Общая кнопка `Бросить все`.

### 2.4 Управление по блокам
- Блок `D20`:
  - Количество кубиков `0..20`.
  - Режим: обычный / преимущество / помеха.
  - Бонус блока: `-20..20`.
  - Кнопка `Бросить D20`.
- Блок `D4-D12`:
  - Для каждого типа кубика количество `0..20`.
  - Общий бонус блока: `-20..20`.
  - Кнопка `Бросить D4-D12`.
- Блок `D100`:
  - Отдельная кнопка `Бросить D100`.
  - Без чекбоксов и без количества.

### 2.5 Общая кнопка
- `Бросить все` бросает только первые 2 блока: `D20` и `D4-D12`.
- `D100` в общий бросок не входит.

### 2.6 Правила блокировки кнопок
- `Бросить D20` активна только при `count_d20 > 0`.
- `Бросить D4-D12` активна только если сумма количеств `d4..d12 > 0`.
- `Бросить все` активна, если активен хотя бы один из первых двух блоков.
- `Бросить D100` всегда активна.

### 2.7 Ввод значений
- Количество и бонусы можно:
  - вводить с клавиатуры,
  - менять стрелками вверх/вниз.
- Значения по умолчанию:
  - все количества = `0`,
  - все бонусы = `0`.
- Значения не сохраняются между открытиями окна:
  - при закрытии/повторном открытии окно сбрасывается в нули.

### 2.8 Логика бросков
- D20 обычный: 1 бросок на кубик.
- D20 преимущество/помеха:
  - 2 броска на каждый кубик,
  - в сумму берется max (преимущество) или min (помеха).
- D4-D12:
  - каждый активированный кубик бросается отдельно,
  - итог = сумма всех результатов + бонус блока.
- D100:
  - единичный бросок `1..100`.

### 2.9 Вывод результата
Показывать блоками результаты только тех групп, которые бросались.

#### D20 (если бросался)
- Мелко: формула, например:
  - `d20:`
  - `3d20(помеха) + 5:`
  - `d20(преимущество) - 4:`
- Крупно: итоговое значение.
- Мелко: значения на кубиках.
- Для преимущества/помехи: для каждого кубика 2 значения, невыбранное полупрозрачное.

#### D4-D12 (если бросались)
- Мелко: формула, например `5d6 + d8 + 5`.
- Крупно: итоговое значение.
- Мелко: значения всех выпавших кубиков.

#### D100 (если бросался)
- Отдельный блок с крупным результатом d100.

### 2.10 Визуал дайсов
В выборе и в выводе использовать фигуры-иконки кубиков:
- d20: правильный шестиугольник (углом вверх)
- d4: равносторонний треугольник (углом вверх)
- d6: квадрат
- d8: ромб (верх/низ 60°)
- d10: квадрат углом вверх
- d12: десятиугольник углом вверх
- d100: как d10 + второй такой же сзади

В центре фигуры:
- в блоке выбора: подпись (`d20`, `d6` и т.д.),
- в выводе: выпавшее число.

## 3) Архитектура модуль-сервиса (изоляция логики)

### 3.1 Подход
Используется изолированный внутренний модуль-сервис, без отдельного процесса и сети.
Это снижает риск поломки существующих окон и упрощает тестирование.

### 3.2 Слои
- `app/services/dice_service.py`
  - Чистая бизнес-логика бросков.
  - Без QML/UI и без доступа к окнам.
- `app/core/dice_controller.py`
  - Состояние окна дайсов.
  - Валидация диапазонов.
  - Вызовы `dice_service`.
  - Подготовка данных для UI.
- `app/ui/qml/DiceWindow.qml`
  - Только интерфейс и отображение.
- `app/core/window_manager.py`
  - Создание/показ окна дайсов по событию.
- `EventBus`
  - `dice.open_requested`
  - при необходимости события для синхронизации состояния.

### 3.3 Границы изоляции
- Dice-модуль не изменяет состояние карты/фона/сцен.
- Dice-модуль не использует существующие визуальные слои map window.
- Ошибки Dice UI/логики не должны влиять на lifecycle других окон.

## 4) План реализации

1. Интеграция окна
- Добавить `DiceWindow.qml`.
- Подключить lazy-open в `WindowManager`.
- Добавить кнопку `Дайсы` в лаунчер.

2. Контроллер и сервис
- Реализовать `dice_service.py` (чистые функции бросков).
- Реализовать `dice_controller.py` (state + команды).
- Привязать controller в QML context.

3. UI-каркас
- Создать структуру из блоков и кнопок.
- Реализовать поле результатов.
- Реализовать stepper-контролы `0..20` и `-20..20`.

4. Логика бросков
- D20 (обычный/преимущество/помеха).
- D4-D12.
- D100.
- `Бросить все` (только первые два блока).

5. Отображение результатов
- Формулы и итоги.
- Детализация значений.
- Полупрозрачные неактивные значения для adv/disadv.

6. Геометрия дайсов
- Универсальный компонент `DieShape`.
- Использование в блоках выбора и в результатах.

7. Сброс состояния
- При открытии окна — полный reset в нули.
- При закрытии/повторном открытии — снова reset.

8. Финальная проверка
- Проверка блокировок кнопок.
- Проверка границ значений.
- Проверка адаптивности на `300x400`.
- Smoke-тест совместимости с лаунчером/картой/фоном.

## 5) Критерии готовности
- Окно открывается из лаунчера без влияния на другие окна.
- Все кнопки бросков работают по ТЗ.
- Формулы и суммы корректны.
- Кнопки блокируются по правилам.
- Состояние сбрасывается при каждом открытии.
- Визуал кубиков соответствует фигурам из ТЗ.
